# Virgil 機能拡張提案書 (Feature Proposals)

本番環境（ローカルLLM環境）での実用性を劇的に高めるための、次期機能拡張の提案です。
Virgil の最大の特徴である「コンテキスト制約（32k-64k）」と「決定論的なオーケストレーション」を念頭に置き、LLMの弱点をツールとシステム設計で補う機能を優先してリストアップしています。

---

## 1. 🔍 セマンティック検索（LanceDB）の本格統合
**課題:** 現在の `search_text` (ripgrep) は強力ですが、「キーワードを正確に推測」できないと該当コードを見つけられません。ローカルLLMは推測が外れて迷走しがちです。
**提案:**
- `GEMINI.md` に記載のある **LanceDB** を活用し、ソースコード群のベクターストア（埋め込み表現）を構築する。
- **新ツール `search_semantic(query="ASTのエラーハンドリング部分")`** を追加。
- **効果:** LLMが「曖昧な自然言語」で関連コードを検索できるようになり、ファイル探索の迷走（不要な `list_files` の連発）を劇的に減らします。

## 2. 🧪 自律的なテスト＆修正ループ (Autonomous Testing Loop)
**課題:** 現在はLLMがコードを書いて終了し、テストの実行やエラー確認はユーザー（または別手順）に依存しています。
**提案:**
- **新ツール `run_tests(test_path="./...")`** の追加。
- エージェントのプロンプトに「コードを変更した後は必ずテストを実行し、失敗した場合はその出力（stderr）を読み込んで自己修正を試みる」というルールを追加。
- **効果:** ユーザーが介入する前に、LLMが構文エラーやロジックのミスを自律的に発見・修正するようになり、成果物の信頼性が跳ね上がります。

## 3. 🗜️ コンテキストの自動圧縮 (Context Summarization & Forgetting)
**課題:** 対話が長引くと、あっという間に Qwen のコンテキスト窓（32k等）を使い切り、APIがエラーになるか極端に遅くなります。
**提案:**
- 古いターン（Turn）をバックグラウンドで要約する処理を導入する。直近 N ターンのみをフルテキストで残し、それ以前は「達成されたタスクの箇条書き」に変換して SQLite に保存し、LLMには要約版のみを渡す。
- **効果:** コンテキスト窓の枯渇を気にせず、1つのセッションで半永久的に作業を継続できるようになります。

## 4. 🌳 AST駆動リファクタリング (Phase 5-2)
**課題:** 厳密マッチ置換（`edit_with_pattern`）は強力ですが、ファイル全体にまたがるリネームや関数の切り出しを一度に行うにはリスクが伴います。
**提案:**
- Tree-sitter の AST（抽象構文木）を直接操作するツールの追加。
    - `rename_symbol(old="Run", new="Execute")`
    - `extract_function(start_line=10, end_line=20, name="newFunc")`
- **効果:** LLM が「文字列の置換」ではなく「プログラム構造の変更」としてコードを編集できるようになり、100%安全なリファクタリングが実現します。

## 5. 🌐 Webリファレンス取得ツール (Web Fetch / RAG)
**課題:** ローカルLLMは学習データが古く（または限定的）、最新のGo仕様や、新しく導入したライブラリのAPIを知りません。
**提案:**
- **新ツール `fetch_docs(url="https://...")`** の追加。HTMLをMarkdownに変換してLLMに読ませる機能。
- **効果:** ユーザーが「このURLのAPI仕様を読んで実装して」と指示するだけで、ローカルLLMでも最新技術を使ったコードが書けるようになります。

---

### 推奨されるロードマップ
まずは実装コストが低く、効果が絶大（エージェントが放置で動くようになる）な **「2. 自律的なテスト＆修正ループ」** と、ローカルLLMの致命的な弱点（最新の知識不足）を補う **「5. Webリファレンス取得ツール」** から着手することをお勧めします。
